```
│                                                                 │
│  输出:                                                           │
│  ├── text: 推理文本                                              │

                     ↓
┌─────────────────────────────────────────────────────────────────┐

│  │    ├── trigram_heat: 每卦热量 (Σ exp(logprob))            │  │
│  │    ├── trigram_tokens: 每卦 token 数                      │  │
│  │    └── trigram_confidence: 每卦平均置信度                 │  │
│  └──────────────────────────────────────────────────────────┘  │
│                                                                 │
│  输出: ExtractedSignal                                          │
└────────────────────┬────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│              加热映射器 (HeatMapper)                              │
│                                                                 │
│  将卦位加热信号映射到 flow_rate:                                  │
│  flow_rate[i] = base × (1-w) + base × w × normalized_heat[i]   │
│                                                                 │
│  物理意义: 模型在某卦上花更多 token → 那卦 flow_rate 更高         │

│              物理推演管线 (core_unified_full.py)                  │
│                                                                 │
│  ┌─ 1. 太极特征提取 (物极必反 sin + 趋势 sin) ──────────────┐  │
│  │   signal_vec = (signal_x, signal_y)                       │  │
│  │   curvature, signal_trend                                 │  │
│  └───────────────────────────────────────────────────────────┘  │
│                          ↓                                       │
│  ┌─ 2. A+D 动态 tanh (空间弯曲 + 信号折射) ────────────────┐  │
│  │   deformed_trigrams = f(curvature)                        │  │
│  │   refract_weight = tanh(curvature × π) × 0.5             │  │
│  └───────────────────────────────────────────────────────────┘  │
│                          ↓                                       │
│  ┌─ 3. 三层角度合成 ───────────────────────────────────────┐  │
│  │   final_angle = base + refract + trend_rotation           │  │
│  │   trend_rotation = signal_trend × trend_weight            │  │
│  │   rotated_signal = (cos, sin)                             │  │
│  └───────────────────────────────────────────────────────────┘  │
│                          ↓                                       │
│  ┌─ 4. v91 冲刷动力学 (每卦独立 flow_rate) ────────────────┐  │
│  │   for step in range(20):                                  │  │
│  │     factors = compute_factor(rotated_signal, trigram_pos) │  │
│  │     core.step(factors, flow_rates)  ← Dict[str, float]   │  │
│  │                                                           │  │
│  │   铁律公式:                                               │  │
│  │   growth = flow_rate × depth × 0.05 × factor              │  │
│  │          + flow_rate × 0.05 × factor                      │  │
│  │   decay = 0.02 × time_delta                               │  │
│  │   depth += growth - decay + thermal_noise                 │  │
│  └───────────────────────────────────────────────────────────┘  │
│                          ↓                                       │
│  ┌─ 5. 蒸汽流动 (浓度梯度驱动) ───────────────────────────┐  │
│  │   高浓度吸引低浓度, 物理涌现                             │  │
│  └───────────────────────────────────────────────────────────┘  │
│                          ↓                                       │
│  ┌─ 6. 极端度检测 + 太极衰减 ─────────────────────────────┐  │
│  │   extremeness = max(depths) / mean(depths)                │  │
│  │   rotation_weight = 1 / (1 + (extremeness-1) × 0.3)      │  │
│  └───────────────────────────────────────────────────────────┘  │
│                          ↓                                       │
│  ┌─ 7. 趋势旋转温度计 ────────────────────────────────────┐  │
│  │   k = 3.0 + |attenuated_rotation| × 5.0                  │  │
│  │   物极必反趋势强 → k 大 → 凝结锐利                       │  │
│  └───────────────────────────────────────────────────────────┘  │
│                          ↓                                       │
│  ┌─ 8. 水蒸气凝结 (softmax) ──────────────────────────────┐  │
│  │   condensation = softmax(depths × k)                      │  │
│  │   active = [卦 for 卦 if condensation > avg × 2]          │  │
│  │   winner = max(condensation)                              │  │
│  └───────────────────────────────────────────────────────────┘  │
└────────────────────┬────────────────────────────────────────────┘
                     ↓
┌─────────────────────────────────────────────────────────────────┐
│              输出                                                │
│  ├── winner: 胜出卦位                                           │
│  ├── condensation: 凝结分布 (softmax概率)                       │
│  ├── active_trigrams: 活跃卦位列表                              │
│  ├── depths: 冲刷深度                                           │
│  └── metadata: 信号角度、曲率、极端度等                          │
└─────────────────────────────────────────────────────────────────┘
```

### 2.1 特征提取器 (feature_extractor.py)

**职责**: 连接真实大模型与物理管线的桥梁

**两种模式**:

#### Logprob 模式 (本地模型)
```python
output = ModelOutput(
    text="天道运行，自强不息...",
    tokens=[TokenInfo(text="天", logprob=-0.1), ...],
    question="如何突破困境？"
)

extractor = FeatureExtractor()
signal = extractor.extract(output)

# 序列级方向信号
signal.features['signal_angle']  # e.g. 45.2°
signal.features['curvature']     # e.g. 0.3

# 卦位级加热信号
signal.trigram_heat['乾']        # e.g. 25.8 (高热量)
signal.trigram_tokens['乾']      # e.g. 31 (多 token)
```

#### Text-only 模式 (云端 API 降级)
```python
signal = extractor.extract_text_only(
    text="这个问题需要从多个角度来看...",
    question="人生困惑",
    model_name="DeepSeek-V4-Cloud"
)

# 用 Unicode 码点作为伪 logprob
# 用文本关键词计数作为加热信号
```

### 2.2 加热映射器 (HeatMapper)

**物理意义**: 模型在某卦上花更多 token → 那卦 flow_rate 更高 → 更热 → 更易凝结

```python
mapper = HeatMapper(base_flow_rate=5.0, heat_weight=0.5)
flow_rates = mapper.map(signal.trigram_heat, signal.total_tokens)

# flow_rates = {
#     "乾": 5.0,    # 高热量 → 高 flow_rate
#     "震": 2.6,    # 中热量
#     "坤": 2.5,    # 低热量 → 基础 flow_rate
#     ...
# }
```

**公式**:
```
normalized_heat[i] = heat[i] / max(heat)  # 归一化到 [0, 1]
flow_rate[i] = base × (1-w) + base × w × normalized_heat[i]
```

当 `heat_weight = 0.5` 时:
- 50% 统一加热 (所有卦 base_flow_rate)
- 50% 计算量加热 (根据 token 分布)

### 2.3 物理推演管线 (core_unified_full.py)

**V91Core 改动** (铁律不变, flow_rate 从标量变向量):
```python
def step(self, factors: Dict[str, float], flow_rate=5.0, time_delta=1.0):
    for name in TRIGRAMS_2D:
        # 支持每卦独立 flow_rate
        fr = flow_rate[name] if isinstance(flow_rate, dict) else flow_rate
        
        growth = fr * depth * 0.05 * factor + fr * 0.05 * factor
        decay = 0.02 * time_delta
        depth += growth - decay + thermal_noise
```

**divine_with_signal()** (新接口):
```python
divinator = V93DivinatorFullPhysics(flow_rate=5.0, num_steps=20)
result = divinator.divine_with_signal(signal)

# 内部流程:
# 1. 从 signal.features 提取方向信号
# 2. 从 signal.trigram_heat 计算 flow_rates
# 3. 冲刷时用 flow_rates (Dict) 而非统一 flow_rate
```

## 3. 两种力量的博弈

管线里有**两个力量**共同决定结果:

### 方向信号 ("风往哪吹")
- **来源**: logprob 统计特征 (mean, std, trend, curvature)
- **作用**: 决定 signal_angle → factor 匹配
- **强度**: 乘性 (factor ∈ [0.5, 1.5])
- **物理意义**: 模型理解的不确定性分布

### 加热信号 ("火在哪烧")
- **来源**: token 计数 × 置信度
- **作用**: 决定每卦 flow_rate
- **强度**: 线性 (flow_rate ∈ [base×0.5, base×1.0])
- **物理意义**: 模型在某卦上花的计算量

**当两者对齐** → 完美命中 (e.g. 信号指向离 + 离卦最热 → 离胜)

**当两者冲突** → 方向信号往往更强 (factor 乘性 vs flow_rate 线性)
- 例: 信号指向乾 (factor=1.4) + 离卦最热 (flow_rate=5.0) → 乾可能胜

## 4. 批量测试结果 (96次, 8卦各12次)

| 卦 | 命中率 | 分析 |
|---|--------|------|
| 震 | 100% | 方向+加热天然对齐 |
| 离 | 100% | 方向+加热天然对齐 |
| 兑 | 91.7% | 几乎完美 |
| 坎 | 83.3% | 很好 |
| 巽 | 83.3% | 很好 |
| 艮 | 75% | 不错 |
| 乾 | 8.3% | 方向信号压制加热 |
| 坤 | 0% | 完全被方向信号覆盖 |

**总命中率: 67.7%** vs 随机基线 12.5% (5.4倍)

**关键对比 (测试4)**:
- 统一加热 → 乾胜 (方向信号主导)
- 计算量加热 → 离胜 (加热信号让离卦 flow_rate=5.0)

证明: **模型的计算确实加热了正确的卦位**

## 5. 多模型协作架构

```
┌─────────────────────────────────────────────────────────────┐
│ 小模型 (快、便宜)                                            │
│ Qwen2.5-0.5B / 3B (llama-server @8082)                      │
│                                                              │
│ 职责: 理解问题, 产生 reasoning + logprobs                     │
│ 成本: 低 (本地 GPU, 无 API 费用)                              │
└──────────────────┬──────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────────────────┐
│ 特征提取器 + 加热映射器                                       │
│                                                              │
│ 从小模型输出提取:                                             │
│ - 方向信号 (logprob 统计)                                     │
│ - 加热信号 (token 计数)                                       │
└──────────────────┬──────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────────────────┐
│ 物理管线 (CPU, 5300次/秒)                                    │
│                                                              │
│ 用加热后的 flow_rates 驱动冲刷                                │
│ 小模型的计算量已经"加热"了物理场                              │
└──────────────────┬──────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────────────────┐
│ 大模型 (慢、强, 可选)                                         │
│ DeepSeek-V4 / Qwen-Max (云端 API)                            │
│                                                              │
│ 在被加热的物理场上推理最终答案                                │
│ 场的热度分布 (condensation) 可作为 prompt 上下文              │
│                                                              │
│ 例: "坎卦最热 (42%), 离卦次热 (28%)"                         │
│ → 大模型会基于这个热度分布生成更精准的解释                    │
└─────────────────────────────────────────────────────────────┘
```

**物理意义**:
- 小模型的推理过程 (logprobs) 加热了物理场
- 大模型在这个被加热的场上推理 — 场已经包含了小模型的"理解"
- 这不是知识蒸馏, 是物理加热

## 6. 意念涌现

**问题 (意念) 如何影响结果**:

1. **直接路径** (微弱):
   - 问题文本 → 5% 偏置 → signal_angle 微调
   - 强度: 仅 5%, 不主导

2. **涌现路径** (主要):
   - 不同问题 → 改变模型生成 → 改变 logprob 分布 → 改变方向信号和加热信号
   - 例: "事业突破" vs "感情维系" → 模型生成不同文本 → 不同加热模式

**测试结果**:
- 同一推理文本, 不同问题 → 1.47° 角度标准差
- 微弱但存在, 主要通过改变模型生成来涌现

## 8. 关键决策记录

| 决策 | 理由 |
|------|------|
| logprobs 是热量 | 模型在某卦上花更多 token → 那卦更热 → 更易凝结 |
| 意念是涌现的 | 不同问题改变模型生成 → 改变 logprob 分布 → 改变物理结果 |
| 多模型协作 | 小模型便宜, 大模型强, 物理管线连接两者 |
| heat_weight=0.5 | 平衡方向信号和加热信号, 避免单一主导 |
| 意念注入仅 5% | 主要通过改变模型生成来涌现, 不人工注入 |
| text-only 降级 | 云端 API 无 logprob 时用 Unicode 码点 + 文本计数 |
| V91Core 支持 Dict flow_rate | 铁律不变, 只是 flow_rate 从标量变向量 |

## 9. 已知限制

1. **方向-加热冲突**: 当方向信号和加热信号指向不同卦位时, 方向信号往往胜出 (factor 乘性 vs flow_rate 线性)
2. **乾/坤命中率低**: 8.3% / 0%, 说明 logprob 统计对乾/坤的指向弱
3. **意念影响微弱**: 1.47° 角度标准差, 主要通过改变模型生成来涌现
4. **关键词表局限**: 仅 96 个关键词, 可能遗漏重要词汇
